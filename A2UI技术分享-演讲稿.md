## 第一部分：启程——前端开发的效率困境（约2.5分钟）— Slide 3-4

### 传统开发模式难以为继

年前2月份的时候，国内营销APP的有一批需求，运营侧有10多个报表页面需求等着上线。

传统流程是这样的：UI设计稿出来，前端手动编码，再和后端联调，**耗时数天，沟通成本高，迭代周期长**。

那后来我们有了AI，有了珊瑚，于是我们先去珊瑚搭建了一个工作流，用AI生成了HTML的页面，最后在APP上呈现出来。

### AI生成页面：一条可行的新路

这是我们开始探索AI赋能开发的新方式——用自然语言描述需求，AI直接生成页面，即刻呈现。

我们选择让AI直接生成HTML，因为它有三个明显优势：**使用简单**——自然语言描述即可生成页面；**输出可靠**——标准化HTML，跨平台兼容；**即时呈现**——分钟级交付，所见即所得。事实上，我们国内营销APP中的**经分体**模块，就是用AI生成HTML的方式实现的，已经跑通了从AI到页面的链路。

**但问题真的解决了吗？**

## 第二部分：瓶颈——传统HTML方案的三大局限（约2分钟）— Slide 5

### 从"能用"到"好用"之间，差了什么？

HTML方案在实际落地中，我们发现了**三个局限**：

1. **交互弱**——纯HTML是静态的，没有数据绑定，无法联动操作，**只能看不能交互**
2. **可靠性差**——不同模型输出风格偏差大，无法统一页面标准
3. **安全隐患**——直接渲染外部HTML，存在XSS注入等安全风险

它们指向同一个本质：**传统HTML模板只能看，我们需要让它"动起来"——用户要能填表单、选选项、点按钮、提交数据，形成完整的交互闭环。**

换句话说，**如何让AI不仅生成"看得见"的页面，还能生成"用得了"的界面？**这才是横在我们面前的真正挑战。

## 第三部分：破局——A2UI协议与项目落地（约7分钟）— Slide 6-20

### A2UI协议：从HTML到JSON Schema

既然HTML方案有局限性，那业界有没有更好的方案？今年3月，参加集团双周会的同事分享了最新的AI前沿知识，其中提到了**A2UI**这个概念。这引起了我的关注，于是我开始深入研究并最终将它落地到项目中。**从概念认知到工程落地，这是我们这次实践的一条主线。**

A2UI全称 Agent to User Interface，是Google推出的开放协议，它是**AI Agent到用户界面的标准化桥梁**。核心理念两句话：**对话即界面**——用户用自然语言和AI对话，AI实时生成可交互的UI；**界面即对话**——界面上的操作直接反馈到AI对话流中，形成闭环。这个理念其实最早来自Google的愿景——**让AI不只是输出文字，而是直接输出可交互的界面**。我们把这个愿景落地到了企业级场景中，形成了A2UI的工程实践。

A2UI的发展经历了五个阶段的演进：从Prompt生成UI雏形，到AI工作流编排，到结构化UI输出（A2UI v0.8首次发布），到生成式应用（v0.9新增多端渲染器），再到Agent驱动完整UI系统。我们正处于从第四阶段向第五阶段迈进的节点上。

业界也有同类探索：**灵光**——融合自然语言理解与全模态交互的AI协作平台；**Google Opal**——用自然语言加可视化流程快速搭建AI小应用。它们与A2UI在技术理念上高度互补，**共同推动AI从"只能输出文字"到"能输出可交互界面"的范式演进。**

### 落地：海豚运维桌面端

有了方案，接下来就是落地。今年3月，我们启动了**海豚运维桌面端**项目。这个项目**关键选型是A2UI Vue Engine——JSON Schema驱动，零代码扩展。**

A2UI在项目中的落地分两个阶段：

**阶段一：场景规则驱动生成**。用户说"我要申请报销"，AI按业务规则自动生成对应的报销申请表单界面，用户填写后提交，数据自动流转到后端系统。

**阶段二：自然语言驱动生成**。用户只需说"我要申请网络权限"，AI自动生成权限申请表单，自动解析表单数据并提交至OA系统。**无需手动填写，一键完成审批流程。**

再看一个实战案例——**对话式运维工单**：运维人员用自然语言描述问题，AI自动识别并生成运维界面，一键提交工单，数据自动流转至工单系统，**无需手动录入**。

### 技术实现：两步走

整个技术实现分两步：

**第一步：页面呈现**。Agent端输出描述页面意图的结构化JSON，而非HTML；客户端接收JSON，Vue渲染引擎完成界面呈现。

**第二步：数据流转**。提前配置解析规则，将用户会话历史交由AI解析；根据配置的处理规则，执行代码传输至IT系统或Agent。**这样就从"看"到了"用"，形成了完整的交互闭环。**

### 技术架构全景

我们项目的技术架构从上到下分五层：

1. **AI输出层**——大模型按A2UI协议生成标准化的UI描述，这是整个项目的一个输入源
2. **协议层**——A2UI协议定义Schema格式与流式传输规范，是大模型与UI之间的桥梁
3. **渲染引擎层**——a2ui-vue-engine，将Schema解析为Vue组件并渲染
4. **组件层**——16个内置组件加自定义扩展，覆盖表单、展示、操作全场景
5. **输出层**——跨端渲染输出，一套Schema多端通用

### 核心引擎能力

a2ui-vue-engine有四大核心能力：

1. **Schema驱动渲染**——用JSON Schema声明式描述UI，引擎自动解析渲染为Vue组件，AI只需生成JSON即可产出界面
2. **双格式加流式协议**——扁平格式（AI友好）与树形格式（渲染友好）自动转换
3. **声明式数据绑定**——支持path、literal、expression三种绑定类型，表单数据自动提取，**这是实现"交互"的关键**
4. **组件映射加可扩展**——16个内置组件通过映射表注册，支持动态注册自定义组件

### JSON Schema vs HTML：为什么这么设计？

传统HTML模板有四大问题：**强耦合、不安全、难扩展、非AI友好**。而A2UI的JSON Schema方案：**声明式、安全、跨端、AI友好**。**本质上，JSON Schema是AI和UI之间的"普通话"——AI说JSON，渲染引擎听懂了，用户就看到了界面。**

### 渲染模式选型

在渲染模式上，Google的A2UI采用流式渲染，首屏响应快但实现复杂、可能出现布局抖动。我们选择了**直接输出模式**——引擎一次性解析渲染，组件间数据绑定完整可靠，渲染结果稳定无抖动。我们的业务场景以表单和操作界面为主，**完整渲染更稳定可靠，这是务实的工程选择。**

## 第四部分：展望——从项目到可复用能力（约2分钟）— Slide 21-26

### 不止于一个项目

A2UI的能力不局限于运维场景。它的核心价值在于：**零代码生成、安全可控、跨端复用**。这意味着，凡是需要"AI生成可交互界面"的场景，都可以用同一套方案：

- **智能审批与流程申请**——权限申请、工单提交、审批流程，AI自动生成表单
- **智能运维**——故障排查、服务重启、日志查看，对话即操作
- **数据分析**——运营看板、报表展示、多维筛选，AI按需生成
- **智能客服**——产品咨询、售后工单、进度查询，交互式界面即时呈现

### 能力封装：让团队零门槛使用

为了让A2UI能力从"我们能用"变成"大家都能用"，我们做了两层封装：

**第一层：npm组件库**——a2ui-vue-engine已发布到npm，任何团队`npm install`即可引入，开箱即用，版本可控，跨项目复用。

**第二层：SKILL封装**——我们制定了SKILL.md规范，包含两种JSON格式、四大组件类、数据绑定规则、布局模式、交互配置和10条最佳实践；同时提供reference.md组件API参考手册，覆盖15个组件、4大分类。**目标是让团队零门槛使用A2UI能力。**
